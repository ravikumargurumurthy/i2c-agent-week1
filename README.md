# I2C Cash Application Agent — Week 1

The cash application step in invoice-to-cash is labor-intensive: AR analysts
read remittance text from emails, PDFs, and EDI files, then figure out
which payments cover which invoices, what's been short-paid, and what
deductions are being claimed. Misallocations cost real money — wrong
customer credited, dispute aged out, write-offs that shouldn't have
happened.

This is a remittance extraction agent for that workflow. Takes
unstructured text, calls three deterministic tools (regex parsing,
fuzzy customer lookup, open-AR verification) via OpenAI tool calling,
and emits a validated `RemittanceAdvice` with confidence-banded
routing for downstream auto-apply / HITL / exception queues.

Built in 5 days as Week 1 of a 90-day Data Engineer → Agentic AI
Engineer transition. Eval suite: **10/10 passing** on a 10-case
hand-labeled set covering clean extraction, short-pay deductions,
ambiguous customers, missing invoices, under/over-allocation, and
unknown payers.

**This is deliberately not a chatbot.** It's a stateful tool-calling
loop designed for finance ops, where accuracy beats fluency,
schema validation is non-negotiable, and uncertain cases route to
humans rather than auto-execute.

## What this should NOT be used for

- **Production cash application.** Synthetic data, single-currency, no real
  ERP integration. Treat as a learning project, not a deployable system.
- **Auto-applying payments at any volume without observation.** The eval
  suite is 10 cases. Real production would need 100+ labeled cases plus
  4+ weeks shadow-mode validation against existing manual processes
  before any auto-execute decisions.
- **Multi-currency or multi-entity remittances.** USD-only; no FX, no
  intercompany handling.
- **PDF, Excel, or EDI input.** Text input only in this version. Multi-format
  extraction is on the Week 4 roadmap.
- **Audit-grade traceability.** `verbose=True` writes to stdout; production
  would require an immutable append-only audit table with model version,
  prompt version, tool calls, intermediate scores, and final decision per
  remittance.
- **Trusting `extraction_notes` as an audit log.** It's a useful reasoning
  trace generated by the model. It is not a substitute for structured
  audit records.

## What this is

The cash application step in I2C is where customer payments get matched
to open invoices. It's labor-intensive: analysts read remittance notes
(emails, PDFs, EDI 820 files) and figure out which invoices each payment
covers, what's been short-paid, and what deductions are being claimed.

This agent automates the extraction step: turn a remittance string into
structured allocations a downstream matching engine can act on.

It's deliberately **not** a chatbot. It's a stateful tool-calling loop
designed for a finance-operations workflow where:

- Accuracy matters more than fluency
- Outputs must validate against a strict schema
- Low-confidence cases route to human review, not auto-execute
- Every decision is reproducible and auditable

## Design decisions worth highlighting

These are documented decisions made over the 5 days, not afterthoughts.

### Deterministic-first reasoning

The LLM never does what code can do better. Money parsing is regex.
Customer lookup is fuzzy matching with suffix stripping. Invoice verification
is table lookup. The LLM only handles ambiguity resolution and final
assembly. Without this discipline, agents hallucinate amounts and
invent invoice numbers.

### Schema is the contract

Every tool result is JSON. Every LLM final output passes Pydantic
validation. Validation failures trigger an automated repair attempt
(max 1 retry) where the error message is fed back to the model.
~70% of malformed outputs recover on the first repair attempt.

### Errors are data, not exceptions

Tool failures (unknown args, missing data, etc.) become structured error
strings the LLM reads on the next iteration. The agent loop never crashes
from a tool error — it self-corrects or gracefully lowers confidence.

### Confidence bands drive routing, not autonomy

The agent doesn't decide whether to apply payments. It produces a
confidence score; the downstream system decides what to do with it
(>=0.95 auto-apply, 0.70-0.94 HITL review, <0.70 exception queue).
This is the correct pattern for finance-grade automation.

### Conventions made explicit

A short-pay where the customer paid $2,450 against a $2,500 invoice
and claims $50 damage is modeled as: `total_amount=2450`,
`amount_paid=2450`, `deduction_amount=50` (informational). This matches
the canonical I2C convention where deductions book separately from cash.
The convention is enforced by `validate_amounts()` and explicit prompt
guidance.

## What this should NOT be used for

- **Production cash application.** This is a learning project on synthetic data.
- **Multi-currency remittances.** Single-currency only; no FX handling.
- **PDF / Excel / EDI input.** Text input only in Week 1; multi-format planned for Week 4.
- **Auto-apply at scale.** Confidence scoring is calibrated only to 10 cases.
  Real production deployment would need 100+ eval cases and shadow-mode validation
  against the existing manual process for weeks before any auto-execution.
- **Explanations the agent can't justify.** `extraction_notes` is a useful
  reasoning trace but is generated by the model — not a substitute for an
  immutable audit log.

## Known limitations and what's next

### Limitations
- Eval cases run single-shot; LLM non-determinism may cause boundary cases to
  flip between runs. Production-grade evals need multi-run statistics.
- `validate_amounts()` business rule is informational at the second retry
  (returns advice with flag rather than raising). Calibration of when to
  block vs flag would require real-data observation.
- Customer master and open AR are JSON files, not a real database. Trivial to
  port to Postgres.
- Audit trail is `verbose=True` stdout, not an immutable log table.

### Week 2-12 roadmap
- Week 2: Rewrite as LangGraph state machine; add LangSmith tracing
- Week 3: Multi-format input — PDF, Excel, EDI 820 sample
- Week 4-5: Project 1 polish — Postgres backend, Streamlit HITL review UI, 50+ eval cases
- Week 6-9: Invoice Matching Agent (Project 2) — the harder problem
- Week 10-12: Exception triage workflow combining both agents
